## Key Findings

- **Novastar protocol**: Reverse-engineered TypeScript library at github.com/sarakusha/novastar — supports TCP (port 5200) and serial. 1000+ API methods. This is the foundation for Phase 4.
- **Phone HDMI output**: Web apps can't programmatically control external displays from phones. iOS/Android just mirror. But getUserMedia (rear camera) works independently while screen outputs HDMI — this enables camera-assisted panel mapping in phone-direct mode.
- **ArUco markers**: js-aruco2 is the right library for Phase 2 panel identification (~80KB, zero deps, handles 12-48 markers).
- **Svelte 5 gotcha**: `$state` rune only works in `.svelte` and `.svelte.ts` files, not plain `.ts`. Store files must be `*.svelte.ts`.
- **PIN auth is a breaking deploy change** (Pass C, 2026-05-31): the server now requires a PIN on every WS connection. A running output box / paired phone must re-pair with the new PIN after a server restart unless `WONDERWALL_PIN` is set to a fixed value on the host. Always set `WONDERWALL_PIN` in the deploy environment so restarts don't silently drop the wall.
- **Render at device pixels, never `ctx.scale(dpr)`** (Pass C): pixel-exact patterns (resolution-check, crosshatch) must render 1:1 in device pixels or they misrepresent on HiDPI/Retina outputs. The `sizeCanvasToDevicePixels` helper (app) and `sizeCanvas()` (output page) own this; don't reintroduce a scale transform.
- **Output-page inline renderers are guarded by pixel parity, not eyeballing** (Pass E, 2026-06-17): `output/index.html` duplicates every pattern renderer so the HDMI box needs no build step. `output-render-parity.test.ts` extracts that inline `patterns` object (via `new Function` on the HTML slice from `const BARS_75` to the end of the `patterns` literal — keep that block contiguous and self-contained, depending only on ctx/Math/the const tables above it) and renders each one on a node-canvas beside its `@wonderwall/patterns` twin, asserting **0** differing pixels. Because node-canvas is deterministic, faithful renderers diff to exactly 0 — so when you add/change a pattern, update BOTH the package and the inline renderer, and use the *unrounded* cell/step dims for derived font sizes and `round(a)+round(b)` (not `round(a+b)`) for label positions, matching the package exactly. Cosmetic differences (color shorthand, `hsl()` spacing) are invisible to the test (same pixels); structural drift (a missing label, wrong color, off-by-rounding text) fails it. `canvas` is a devDep of both `patterns` and `server`.
- **A socket `'close'` handler must scope itself to its own socket** (Pass F, 2026-06-17): `connectToDevice` keeps one module-level `socket`/`session`/`connection`. Reconnecting destroys the previous socket, but its `'close'` fires *after* the new socket is installed, so a handler that nulls the shared state unconditionally clobbers the new connection (manifests as a 5s `CONNECT_TIMEOUT` hang on the second connect). The fix pattern: capture the socket in a local `const sock`, set `socket = sock`, and gate every shared-state mutation in its `'error'`/`'close'`/connect callbacks behind `if (socket === sock)`. Same applies to any future per-connection lifecycle callback that touches module state — bind it to the local socket, not the module slot.
- **Param bounds are enforced by `sanitizeParams`, not by the UI** (Pass D, 2026-06-17): the declared `min`/`max` on a `PatternParameter` are the contract; `sanitizeParams(pattern, raw)` is the *only* thing that enforces them. It runs at two chokepoints — the patterns package's render entry points (whole app) and the server `setPattern` relay (output page + controllers). The UI sliders clamp too, but the WS path and corrupted localStorage do not, so never rely on the UI for safety. A param that drives a `+= step` loop (crosshatch/resolution/motion-test) must also be intrinsically clamped inside the pattern (`Math.max(1, …)`) so a direct library call can't hang. Invariant: `sanitizeParams(p, getDefaultParams(p))` must deep-equal the defaults for every pattern (there's a test) — if you add a pattern whose default sits outside its own `min`/`max`, that test will catch it.
- **A localStorage `loadFromStorage` must validate shape, not just survive `JSON.parse`** (Pass G, 2026-06-17): the persisted stores (`wall.svelte.ts`, `presets.svelte.ts`) seed `$state` directly from localStorage at construction. Wrapping `JSON.parse` in try/catch only handles *malformed* JSON — valid JSON of the wrong shape (`"null"`, `{}`, an array, a stale-schema object) parses cleanly and then crashes a getter (`null.columns`) or a list op (`[...nonArray]`, `.filter`). The pattern: parse to `unknown`, run a type guard (`isValidDimensions`/`isPreset`), and fall back to a safe default (`Array.isArray(p) ? p.filter(isPreset) : []`). The guard must accept everything the store's own `set()`/`save()` writes (round-trip safe) — verify against the writer. This mirrors the `sanitizeParams` philosophy on the *persistence* boundary: the write side was already guarded (Pass C), the read side wasn't.
- **An async singleton acquired before teardown must detect that it lost the race** (Pass H, 2026-06-18): `services/camera.ts` is a module-level singleton, and `startCamera` is async (`getUserMedia` awaits a permission prompt). The consuming views (`CameraMapper`, `DiagnosticsRunner`) call `await startCamera` on mount and `stopCamera()` on unmount/cancel — so a teardown can land *while acquisition is still pending*, and a naive `stream = await getUserMedia()` then installs a resource nobody will release (camera/privacy light stuck on). Fix pattern: a monotonic `generation` token bumped by every `stopCamera()`/`startCamera()`; after the await, `if (myGeneration !== generation) { release(); return; }`. Two windows must both be covered — the **getUserMedia await** (handled by the token, before assignment) and the **`video.play()` await** (handled because the stream is already assigned, so `stopCamera` finds and stops it). Load-bearing invariant: **no `await` between the generation check and the assignment**. Any future async-acquire singleton (camera, socket, wake-lock) needs the same guard. Also: a component's `catch` around `await startCamera` must `stopCamera()` — `getUserMedia`/`play()` can reject *after* the device was acquired.
